# WorkPortal 基本設計（権限・表示ルール）

このドキュメントは WorkPortal の**権限・表示の確定仕様（基本設計）**です。
2026-06-26 時点で実装・検証済みの内容を基準とします。

## 運用ルール（重要）
- 新しい改修命令が、この基本設計と**異なる／矛盾する**場合、実装前に必ず
  「**基本設計と異なる　よろしいですか？**」と確認すること。
- ユーザーが承認した場合のみ実装し、**この基本設計に追記・更新**する（末尾の「変更履歴」にも1行記録）。
- 基本設計に無い新規項目の追加は、確認のうえこのドキュメントへ追記する。

---

## 1. 認証・セッション
- 全ページ・全APIは**署名付きセッション必須**。Cookie `workportal_auth` は HS256 署名トークン（httpOnly）。
- 表示用に `workportal_user`（ユーザー名・非httpOnly）を別途保持。**認証には使わない（信頼しない）**。
- middleware が署名・有効期限を検証。未認証は **ページ＝SSOへリダイレクト／API＝401**。
- 検証済みユーザー名のみ信頼ヘッダ `x-wp-user` で下流へ。クライアント偽装ヘッダは除去。
- ログインは orgportal SSO 経由（既定 24時間保持。期限内はブラウザを閉じても自動ログイン）。
- ログアウトは `/api/auth/logout`（両Cookie破棄＋orgportalシングルログアウト）→ `/login`。

## 2. 役割の定義
- **取締役** = `isDirector` または `isRepresentative`（代表取締役）。承認・全閲覧権限の基準。
- **顧問** = `isAdvisor`。取締役会メンバーだが**承認権限なし**（権限上は社員相当）。
- **取締役会メンバー** = 取締役・代表取締役・顧問。
- **部長** = `position === 'manager'`。「担当部長」= 対象部門の部長（部門一致）。
- **課長** = `position === 'chief'`、**社員** = `position === 'staff'`。
- **管理者** = `role === 'admin'`（システム管理）。

## 3. 表示範囲（可視性）
- 既定ビュー：取締役以上＝**全体**／部長＝**自部門**／社員・課長＝**自分**。
- 決定事項・実行タスク・プロジェクト一覧：取締役＝全件／部長＝自部門に関連／社員＝自分が関与。
- **全社通達**（決定の `departmentId === 'all'`）＝**全員に表示**。
- **取締役会限定**（`boardOnly`）＝取締役会メンバー＋該当する担当部長のみ表示・作成。
- **方針ビジュアル**（/projects ビジュアルタブの方針ノード）＝**全員が全方針を閲覧可**。
  - ただし方針配下のプロジェクト・決定・タスクは従来どおり関与ベースで絞り込み。
  - **管理タブ**の方針一覧・権限は変更しない（従来スコープ）。
- 承認履歴（/history）：取締役会＝全件／部長＝自部門＋本人関与（**方針・PJは本人関与のみ**）／課長・社員＝本人関与のみ。
  - 「本人関与」= 決定:起案者/担当者/タスクwho/承認者、方針PJ:起案者/承認者のみ。

## 4. 承認マトリクス（方針・プロジェクト・決定事項）

**階層構造**：**方針 ＞ プロジェクト ＞ 決定事項 ＞ 実行タスク** の4階層。/projects のビジュアルタブでこの順にドリルダウン表示する（方針ごとのプロジェクト数→決定事項→タスクを展開）。方針タグ有り・プロジェクト未指定の決定は各方針配下に「（プロジェクト指定なし）」として表示。

**ビジュアルの受け皿ノード（実在レコードではない・表示専用）**：
- **「（方針未割当）」**＝方針(policy)が未設定のプロジェクトをまとめる受け皿。**実在の方針ではない**ため**管理タブの方針一覧には出ない**（正しい挙動）。配下のプロジェクト自体は管理タブの「プロジェクト」一覧に（方針空のまま）存在し、各プロジェクトの「方針」プルダウンで割り当て可能（担当部長＋取締役・割り当ては再承認）。
- **「（プロジェクト指定なし）」**＝方針には紐づくがプロジェクト未指定の決定事項をまとめる受け皿（各方針配下）。
- いずれも表示上のグループ見出しで、DBレコードではない。

**承認できる画面**：方針・プロジェクトの承認は **/projects の「管理」タブと「ビジュアル」タブの両方**から可能（権限者のみ承認ボタン表示。承認待ちは進捗 n/2・あと◯名を表示）。

### 方針（Policy）
- 作成：**取締役のみ**
- 編集：**取締役のみ**（編集すると再承認）
- 承認：**取締役2名**（別々の人）
- 削除：削除申請→**取締役2名**承認で削除。直接削除API＝取締役のみ
- ビジュアルは全員閲覧可（§3）。

### プロジェクト（Project）
- 必須項目：**名称・説明・部門・担当者**（作成・編集とも空白不可）
- 作成：**担当部長（対象部門の部長）＋取締役**（部長は自部門のみ）
- 編集（名称・説明・部門・担当者）：**担当部長＋取締役**（編集すると再承認）
- 方針の付け替え／並び替え：**担当部長＋取締役**、かつ**再承認（取締役2名）**
- 承認：**取締役2名**（別々の人）。**部長は承認不可**
- 削除申請：**担当部長＋取締役**（実削除は取締役2名承認。担当部長の申請は人数に数えない）。直接削除API＝取締役のみ
- 画面：追加フォーム・編集・方針付け替え・削除申請は権限者のみ表示。一覧・カードに**部門・担当者を表示**。
- 担当者は**全メンバーから選択**（初期値＝入力者本人）。部門は取締役＝全部門選択可／部長＝自部門固定。
- **既存で部門・担当者が空のプロジェクト**は「未設定」と表示（一括バックフィルはしない）。編集時に必須入力を求める。

### 決定事項（Decision）
- 作成：ログイン者（担当部署で起案）。取締役会限定は取締役会メンバーのみ
- 編集：**入力者(起案者=createdBy)＋担当者(assigneeUsername)＋担当部長＋取締役**（編集すると再承認）
- 承認：**担当部長1名＋取締役1名**（別々の人）。**全社通達・取締役会限定は取締役2名**
- 削除申請：**入力者＋担当者＋担当部長＋取締役**（実削除は上記承認要件）。直接削除API＝同条件
- **全社通達(部門=all)の「担当部長」＝担当者(assignee)の部署の部長**とみなす（通常は決定の部門の部長）。

#### 決定事項：表示・動作
- カードに **担当部門（「担当」）／担当者／起案者（「起案」）** を表示。承認待ちは **承認進捗 n/2・あと◯名**＋承認者名（取締役/担当部長）。自分が承認済みなら「✓承認済み」化（二重承認不可）。
- バッジ：全社通達＝**📢全員通達**、取締役会限定＝**🔒取締役会限定**、状態＝承認待ち/承認済み/完了。
- 編集すると承認リセット＝**再承認（pending）**。承認待ち中は **EditDiff** で変更内容（before→after・全項目）を表示。
- **編集取消**：承認待ちの間のみ・**編集した本人**のみ（編集前スナップショットに復元）。
- **承認取消**：自分の承認・承認待ち（2名未達）・**承認から30分以内**。
- **後追いタスク追加**：**部長以上（部長・取締役・管理者）**のみ。決定事項カードからの追加は**再承認**（追加タスクは承認まで非表示=pendingEdit）。実行タスクページからの追加は承認不要・即稼働。
- タスク0件かつ承認済みの決定は手動で**「完了／未完了」切替**（担当部長＋取締役・画面で制御）。タスクがある決定は**全タスク完了で自動「完了」**。

## 5. 実行タスク・個人タスクの編集/削除

### 実行タスク（決定事項由来・全件共通）
- 編集・ステータス変更（未着手/進行中/完了）：**入力者(起案者=createdBy)＋担当者(who)＋担当部長＋取締役**
- 削除：**担当者(who)＋担当部長＋取締役**（※**起案者は編集はできるが削除は不可**）
- `createdBy`（入力者）は**作成時のみ設定。後から決定に紐づけても上書きしない／紐づけ者を作成者にしない**。
- **ステータス変更（進捗：未着手/進行中/完了）は再承認不要**。内容（5W1H・タグ）編集は**そのタスクのみ**親決定を再承認（pendingEdit）。編集取消は本人のみ（承認待ち中）。
- /todos の表示は**2段階展開**：①タスク→決定事項＋**決定事項作成者** ②「5W1H（実行タスク）」→ 5W1H＋**実行タスク作成者**＋編集（決定事項は残したまま）。方針/プロジェクトタグを付与・表示可。

### 個人タスク（Todo・共有含む）
- 編集・ステータス変更・削除：**担当者(所有者 userId)＋担当部長(所有者の部署の部長)＋取締役**
- 共有(isShared)タスクも同じ権限。編集で担当/部門を編集者にすり替えない（所有者のまま）。
- 項目：5W1H（開始/完了予定/なぜ/どこで/どうやって/担当/部門）＋優先度＋公開設定(isShared)。工程(steps)はUI廃止。

### 共有タスク（/todos の「共有」タブ）
- 共有タブに出るもの＝**共有(isShared)の個人タスク**＋**全社通達の実行タスク（担当が自分でない＝自分タブに出ないもの）**。
- 全社通達の実行タスクは、who＝自分なら自分タブにも表示（重複集約のルール）。
- ダッシュボードの期限超過から来た時、自分タブに超過が無くても共有側に超過があれば
  「**共有タスクに（も）期限超過タスクがあります**」と案内バナーを表示。

## 5B. ダッシュボード（/dashboard）
ダッシュボードの各集計は、**ユーザーが /todos・/decisions で見られる範囲と一致**させる（`defaultView`：取締役＝全体／部長＝自部門／社員＝自分関与）。※過去に「ダッシュボードだけ全社集計」になっていた不具合を是正した重要ルール。期限超過の判定は §7（完了予定日 15:00 JST 超過）。

### 見た目（確定）
- 先頭の挨拶「おはようございます」は**表示しない**（日付のみ）。
- 「今日やること（リアルタイム）」の見出し・更新時刻表示は**出さない**（各枠は従来どおり30秒自動更新・no-store）。
- ダッシュボードの**強調色は藍(indigo)に統一**＝黄色(amber)のマーク（セクションの色ドット・件数バッジ・進行中の点・承認待ちバッジ等）は使わない。承認ボタンも藍。

### 上部「今日やること」（MyWork・30秒ごと自動更新／取得は no-store）
- **あなたの承認待ち**：自分が承認すべき方針・プロジェクト・決定事項（**status=pending のみ**）。承認進捗 n/2・あと◯名、その場で承認可。
- **取締役会限定**：該当時のみ表示（取締役会メンバー＝全件／担当部長＝本人 or 自部門のみ）。
- **自分の未完了タスク**：自分の個人タスク（未完了）＋自分が担当(who)の決定タスク（未完了）。0件のとき 🏆「全タスク制覇！」。
- **自部門の未完了タスク**：自部門の決定タスク（未完了）。
- 各リストは10件上限＋「すべて見る」。

### KPIカード
- **期限超過（タスク）**：個人タスク（自分/共有・未完了）＋表示範囲内の決定タスク（未完了）で 15:00 超過の件数。クリックで `/todos?overdue=1`。カードに「完了予定日の15:00を過ぎると超過になります」を表示。
- **期限超過（決定事項）**：表示範囲内の決定事項（未完了）で 15:00 超過の件数。クリックで `/decisions?overdue=1`。
- **完了実行タスク**：個人タスク＋決定タスクの `completedAt` を **本日／今週（月曜始まり）／今月／今年**で集計（JST境界）。
- **完了決定事項**：status=done の決定を `completedAt` で同期間集計。
- いずれも集計母数は上記スコープ（自分が見える範囲）。導入前に完了したもの（completedAt 無し）は集計外。
- **完了カードはクリック可能**：完了実行タスク→`/todos?status=done&period=<本日/今週/今月/今年>`、完了決定事項→`/decisions?status=done&period=…`。遷移先は **completedAt がその期間内の完了分**だけを表示（JST境界・§7同様）。
  - ※/todos は既定「自分」タブのため、共有(他人所有)の完了個人タスクは「共有」タブ側で確認する。

## 6. 作成者・起案者の表示
- 決定事項：起案者 = `createdBy`（「起案」表示）。担当部署 = `departmentId`（「担当」）、担当者 = `assigneeUsername`（「担当者」）。
- 方針・プロジェクト：作成者は**「起案者」**と表記。
- 実行タスク：**実行タスク作成者（入力者）**を表示。決定由来タスクは**決定事項作成者＋実行タスク作成者の両方**を表示。
- 個人タスク：作成者 = 所有者。

## 7. 日付・期限（日本時間）
- 日付判定・日時表示はすべて **JST（Asia/Tokyo）**。
- **期限超過 = 完了予定日の 15:00（JST）を過ぎたら超過**（`isOverdueDue`）。
- 「今週締切」は超過分を除外。
- DB格納の入力日時（createdAt 等）の**値は変更しない**（表示のみJST）。

## 8. UI 表示ルール
- 権限の無いユーザーには、編集・削除・追加・承認の**ボタンを隠す/無効化**する（サーバーでも同権限で拒否＝二重防御）。
- カレンダーは**未実装**のためナビ無効（クリック不可・薄表示「準備中」）。メニュー順は最後＝**方針・プロジェクトの下**に配置。
- ダッシュボード「あなたの承認待ち」は **status === 'pending'** のものだけ表示（承認済みは即時に消える）。
- ダッシュボード期限超過カードに「完了予定日の15:00を過ぎると超過になります」を表示。
- 「自分の未完了タスク」が0件のとき 🏆「全タスク制覇！」を表示。
- 失敗（保存・承認等）は**握りつぶさず画面にエラー表示**する。

## 9. データ保全・運用（補足）
- 本番とローカルは同一 Azure SQL を共有。スキーマ変更（db push）は本番即反映＝要承認。
- API取得は `cache: 'no-store'`（承認状態などを古く表示しない）。

## 10. 承認履歴（/history）の表示・記録
可視範囲は §3 を参照（取締役会＝全件／部長＝自部門＋本人関与・方針PJは本人関与のみ／課長・社員＝本人関与のみ）。本節は**表示内容と記録方式**を定める。

### 記録方式（AuditLog）
- 操作のたびに**追記専用の AuditLog** に記録：`approve`（承認）／`delete_approve`（削除承認）／`deleted`（削除実行）／`edit`（編集申請）。
- 記録時に**スナップショット**（対象名・申請者・申請日時・変更内容 before→after・関係者）を保存するため、**対象が削除された後も履歴は残る**。
  - 可視範囲スナップショット列：`deptIds`（部門）/`involved`（本人関与）/`assignee`（この内容の担当者）/`creator`（作成者）。
  - 既に削除済みのエンティティ由来の過去ログは関係者が埋まらない場合がある＝**今後分から正確**（既知の制約）。

### 行（一覧）に表示する項目
- **申請種別バッジ**（先頭）：`新規申請を承認` / `編集申請を承認` / `削除申請を承認` / `編集申請` / `削除を実行`。
- **作成: 〇〇**（`creatorNames`。方針・PJは「起案者」に相当）。
- **担当: 〇〇**（`assigneeNames` = この内容の担当者：決定の担当者＋実行タスクの who）。
- **申請: 氏名（部署）**（approve/delete_approve は申請者、edit は実行者本人）。
- **承認: 氏名（部署）**（approve / delete_approve のとき）。

### 詳細（行クリックで展開）に表示する項目
- 申請種別 / 対象 / **作成者** / **この内容の担当者** / 申請者 / 申請日時 / 承認者 / 承認日時 / **変更内容（before→after の全項目）**。
- 承認者欄には役職（取締役 / 担当部長）も併記。

### ラベル・用語
- 方針・プロジェクトの作成者は **「起案者」** と表記（決定事項は「担当」「起案」）。
- 実行タスク編集の変更内容は「実行タスク『名称』の〈項目〉」形式で、どのタスクの編集か分かるようにする。

### 更新
- **30秒ごと**の自動更新＋**ウィンドウfocus時**に再取得（承認直後の反映遅延を防ぐ）。
- 取得は `cache: 'no-store'`。
